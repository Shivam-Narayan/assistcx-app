
### Overview

AssistCX follows a **Role-Based Access Control (RBAC)** model to manage user permissions across modules and features.

RBAC ensures:
- Users access only what their role allows
- Sensitive operations are restricted
- Platform governance remains controlled

---

### Roles

| Role | Description |
|------|------------|
| **Root** | Super-user with implicit access to all routes. Used by Aexonic during client implementation. Can update permissions, update roles, create agents, create intents, create data templates, etc. |
| **Account Admin** | Full access across the platform (all modules and management capabilities). |
| **System Admin** | High-level access including viewing all modules and creating configurations. Restricted from certain account-level administrative actions. |
| **Group Admin** | Operational-level admin access (view agents, tasks, attachments, mailbox pollings). |
| **Group Staff** | Basic operational access (dashboard, tasks, attachments, profile). |
| **Public User** | Limited access primarily for inbox, tasks, and profile-related operations. |

---

### Modules Covered

### How RBAC Works

#### 1. Role Assignment
Users are assigned roles based on responsibilities.  
Each role has predefined permissions which can be customized if required.

#### 2. Access Control
Access to modules, endpoints, and features is strictly controlled based on assigned roles.

#### 3. Module Interaction
Each role interacts differently with modules:
- **Account Admin** → Full platform control  
- **System Admin** → Configuration + operational visibility  
- **Group Admin** → Limited operational management  
- **Group Staff** → Essential task-level access  
- **Public User** → Minimal functional access

>  **Important Notes**

- Root access is reserved strictly for **platform-level control**
- Organization-level governance is handled by **Account Admin**

- Public users are intentionally restricted for security and scope control
